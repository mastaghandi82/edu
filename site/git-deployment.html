Git & Deployment

Von der lokalen Datei zur Live-URL

Was zwischen git init und einer öffentlich erreichbaren Adresse passiert. Aufgeschrieben beim ersten eigenen Durchlauf — inklusive der Stellen, an denen es hakte.

Diese Seite dokumentiert den Prozess, mit dem sie selbst veröffentlicht wird. Jede Änderung an dieser Datei löst genau die Kette aus, die hier beschrieben steht. Die Doku kann damit nicht veralten, ohne dass es auffällt — sie ist der Testfall.

Die Kette

git push origin main
  │
  └─▸ GitHub Actions startet eine frische Ubuntu-VM
        │
        ├─▸ checkout ............ Repo holen
        ├─▸ upload-artifact ..... Ordner site/ einpacken
        └─▸ deploy-pages ........ veröffentlichen
              │
              └─▸ https://mastaghandi82.github.io/edu/

Niemand lädt Dateien irgendwohin. Ein Push ist der einzige Auslöser. Der Wert liegt in der Reproduzierbarkeit: Die VM ist jedes Mal neu und leer. Es gibt kein „läuft bei mir“ — was nicht im Repo steht, existiert nicht. Eine vergessene lokale Datei wird so sofort zum sichtbaren Fehler statt zur stillen Zeitbombe.

Git in einem Bild

Working Directory  ──add──▸  Staging Area  ──commit──▸  Repository
(deine Dateien)              (Vormerkung)               (Schnappschuss)

Die Staging Area ist der Schritt, den andere Systeme nicht haben, und zuerst wirkt er überflüssig. Sein Nutzen zeigt sich, sobald zehn Dateien geändert sind, aber nur drei inhaltlich zusammengehören: git add nimmt gezielt diese drei. Ein Commit soll eine Änderung sein — nicht „alles, was gerade rumlag“.

Und das Repository liegt vollständig lokal. Die Versionierung ist nach git init sofort komplett da, ohne jeden Server. GitHub ist nur eine Kopie — für Backup, Zusammenarbeit, CI und Hosting. Das ist gemeint, wenn Git „verteilt“ genannt wird.

Vier Fallen beim ersten Mal

1. Der Branch heißt master

git init ohne Argument nennt den ersten Branch master. Der Deploy-Workflow lauscht aber auf main. Folge: kein Fehler, keine Meldung — es passiert einfach nichts. 

git init -b main
git config --global init.defaultBranch main   # ein für alle Mal

2. Die E-Mail landet öffentlich in jedem Commit

Autor-Name und E-Mail stehen dauerhaft in der Historie; in einem öffentlichen Repo lesen Crawler sie aus. GitHub stellt dafür eine noreply-Adresse bereit — die Zuordnung zum Account funktioniert trotzdem, sie läuft über die Adresse, nicht den Namen. 

git config --global user.email "<id>+<user>@users.noreply.github.com"
# eigene ID:  gh api user --jq .id

3. Der workflow-Scope fehlt

Ohne ihn verweigert GitHub das Pushen von Dateien unter .github/workflows/ — mit einer Meldung, die nicht danach klingt. gh auth login setzt ihn mit.

4. Grüner Workflow, trotzdem 404

Die häufigste Falle überhaupt. Der Workflow im Repo aktiviert GitHub Pages nicht. Die Quelle muss einmalig im Browser umgestellt werden: Settings → Pages → Build and deployment → Source → GitHub Actions.

Deshalb ist ein grüner Haken kein Beweis. Nur die aufgerufene URL ist einer.

Ignorieren: zwei Wege, ein Unterschied

.gitignore.git/info/exclude
Wird committetjanie
Aussage„gehört in niemandes Repo“„will ich hier nicht sehen“
Beispielnode_modules/, .enveigene Notizen

Prüfen, warum etwas ignoriert wird — die Ausgabe nennt Datei und Zeile der Regel: 

git check-ignore -v .claude/progress.md
# .git/info/exclude:9:.claude/progress.md

Ignorieren wirkt nur auf noch nie committete Dateien. Was einmal drin war, bleibt drin — die Regel kommt zu spät. Deshalb .gitignore vor dem ersten Commit anlegen. Bei versehentlich committeten Secrets reicht Löschen nicht: Sie stehen in der Historie und sind als kompromittiert zu behandeln.

Bei Secrets: Allowlist statt Denylist

Wenn in einem Verzeichnis etwas Geheimes liegt, ist die Denylist die falsche Richtung. Nicht auflisten, was raus soll — auflisten, was rein darf: 

# Alles raus...
*
# ...gezielt rein.
!.gitignore
!CLAUDE.md
!config/
!config/**

Der Unterschied ist Sicherheit, nicht Geschmack: Bei einer Denylist ist jede künftige Datei standardmäßig drin — ein Update legt einen neuen Token-Cache an, und er wird stillschweigend mitcommittet. Bei einer Allowlist ist alles Neue standardmäßig draußen. Bei Secrets ist die Default-Antwort „nein“.

Der Workflow

on:
  push:
    branches: [main]

permissions:
  contents: read      # checkout darf lesen
  pages: write        # darf deployen
  id-token: write     # darf ein OIDC-Token anfordern

concurrency:
  group: "pages"
  cancel-in-progress: false

Der permissions-Block ersetzt die Standardrechte vollständig — der Job kann nur, was dort steht. id-token ist das unintuitivste: Damit fordert der Lauf ein kurzlebiges Token an, mit dem Pages prüft, dass das Deployment aus legitimer Quelle stammt. Kein dauerhaftes Secret im Spiel.

cancel-in-progress: false ist bewusst gewählt: Ein laufendes Deployment wird nie abgebrochen. Bei CI-Tests wäre true richtig — veraltete Läufe abwürgen spart Zeit. Ein abgebrochener Produktiv-Deploy hinterlässt dagegen womöglich einen halben Stand.

Versionsnummern nachschlagen, nicht erinnern. Aktuell sind checkout@v7, configure-pages@v6, upload-pages-artifact@v5, deploy-pages@v5 — gegen die GitHub-API geprüft. Kurios: GitHubs eigene Starter-Vorlage zeigt noch einen älteren Mix. Auch offizielle Doku veraltet.

Läufe beobachten — und die TTY-Falle

Am Terminal genügt gh run watch. In einem Skript nicht: 

run ID required when not running interactively

Der Befehl fragt sonst interaktiv nach, welcher Lauf gemeint ist — in einer Pipeline antwortet niemand. Die ID also selbst besorgen, und zwar die des eigenen Commits: 

RID=$(gh run list --commit "$(git rev-parse HEAD)" --limit 1 \
        --json databaseId --jq '.[0].databaseId')
[ -n "$RID" ] || { echo "Kein Lauf gefunden." >&2; exit 1; }
gh run watch "$RID" --exit-status --compact

--exit-status ist nicht optional: Ohne das meldet ein roter Lauf im Skript Erfolg. Und ohne --commit nimmt --limit 1 den neuesten Lauf im Repo — bei parallelen Pushes also fremde Arbeit.

Die tückischere Falle steckt im Fix: --commit verlangt den vollen SHA. Mit einem Kurz-SHA gibt es keinen Fehler, sondern eine leere Antwort — und gh run watch "" läuft in ein nichtssagendes 404. Merksatz: Ein leeres Ergebnis ist kein Beweis für „nichts da“ — es kann auch eine falsch gestellte Frage sein.

Das verallgemeinert sich: CLI-Werkzeuge verhalten sich anders, sobald kein Mensch am Terminal sitzt. Wer Befehle interaktiv testet und dann in ein Skript kopiert, läuft genau hier auf. Dieselbe Klasse: git rebase -i, gh auth login, jedes --interactive.

Der PR-Zyklus — und warum er kein Formular ist

git switch -c thema     →  Branch: hier darf es kaputt sein
git commit              →  festschreiben
git push -u origin ...  →  hochladen
gh pr create            →  zur Prüfung stellen
[ Review durch jemand anderen ]
merge                   →  löst automatisch den Deploy aus

Warum ein Branch statt direkt auf main? Weil main jederzeit deploybar bleiben muss — hier sogar wörtlich: Jeder Push auf main geht live.

Der lehrreichste Moment des ersten Durchlaufs war ein blockierter Merge. Dieselbe Instanz hatte den PR geschrieben, sich selbst reviewt und wollte sich selbst durchwinken — eine Schutzschicht verweigerte das. Zu Recht: Genau das soll ein PR verhindern.

Ein Pull Request ist ein sozialer Mechanismus, kein technischer. Sein Wert kommt aus dem zweiten Augenpaar. Automatisierung darf den Kontrollpunkt nicht mit-automatisieren — wer schreibt, prüft und freigibt, dessen Prüfung ist Theater.

Was hängen bleiben soll

  • Ein grüner Haken ist kein Beweis. Nur die aufgerufene URL ist einer.
  • Versionsnummern nachschlagen, nicht erinnern. Auch offizielle Doku veraltet.
  • Was committet ist, ist fast immer rettbar. Was nie committet wurde, nicht.
  • Bei Secrets ist die Default-Antwort „nein“. Denylists vergessen, Allowlists nicht.
  • Das Warum steht nirgends. Was gemacht wurde, zeigt der Diff. Warum es so entschieden wurde, muss in die Commit-Message — sonst ist es weg.
