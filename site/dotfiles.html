Dotfiles

Konfiguration ist Wissen —
und Wissen braucht ein Repo

Shell-Aliase, Editor-Einstellungen, die persönliche CLAUDE.md: destillierte Arbeitserfahrung, die typischerweise unversioniert auf genau einer Platte liegt.

Warum überhaupt

Für Konfiguration gilt derselbe Satz wie für Chat-Verläufe: Wissen, das nur an einem Ort existiert, ist bereits verloren. Der Plattentod macht es nur schlagartig sichtbar. Ein Repo löst drei Probleme auf einmal: Backup (der offensichtliche Teil), Sync (neuer Rechner: klonen statt nachbauen) — und das Warum: „Warum habe ich diese Regel im März geändert?" steht in der Commit-Message. Sonst nirgends.

Dotfiles heißen die Konfigurationsdateien im Home-Verzeichnis, weil sie mit einem Punkt beginnen (.gitconfig, .zshrc) — Unix blendet sie standardmäßig aus. Das Muster umfasst längst auch ganze Verzeichnisse wie ~/.claude/.

Die entscheidende Sortierung

Der häufigste Fehler ist, zu viel zu versionieren. Im Home liegen drei Sorten Daten — und nur eine gehört ins Repo:

KategorieBeispieleIns Repo?
Konfiguration & WissenCLAUDE.md, skills/, .zshrc, .gitconfigja
Zustand & VerlaufSession-Daten, Caches, Historynein — flüchtig, teils sensibel
GeheimnisseTokens, Keys, .credentials.jsonniemals

Zustand zu sichern ist wertlos (veraltet in Minuten), Geheimnisse zu sichern ist gefährlich. Übrig bleibt genau das, was man nach einem Plattentod wirklich vermisst.

Der Einstieg: Verzeichnis direkt als Repo

Für einen abgegrenzten Ordner wie ~/.claude/ der einfachste Weg. Der Kern ist eine .gitignore nach dem Allowlist-Prinzip — alles verboten, Gewolltes einzeln freigeschaltet: 

/*
!.gitignore
!CLAUDE.md
!settings.json
!skills/
!agents/

Denylists vergessen, Allowlists nicht.

Wer nur .credentials.json auf eine Verbotsliste setzt, committet irgendwann die nächste Datei, an die niemand gedacht hat.

Der Ablauf — mit dem Kontrollpunkt an der richtigen Stelle:

cd ~/.claude
git init
# .gitignore anlegen (siehe oben)
git status     # ← DER Kontrollpunkt: nur Gewolltes sichtbar?
git add . && git commit -m "Claude-Konfiguration versionieren"
git remote add origin git@github.com:<user>/claude-config.git
git push -u origin main

Das git status vor dem ersten add ist kein optionaler Schritt: Hier wird die Allowlist bewiesen — nicht erst im Push.

Verstreute Dateien: das Bare-Repo

Liegen die Dateien quer übers Home (.zshrc, .gitconfig, .config/…), macht man das Home nicht direkt zum Repo — sonst meldet jedes git status überall das Dotfiles-Repo. Der verbreitete Trick: ein Repo ohne eigenes Arbeitsverzeichnis, das Home nur auf Zuruf benutzt: 

git init --bare ~/.dotfiles
alias config='git --git-dir=$HOME/.dotfiles --work-tree=$HOME'
config config status.showUntrackedFiles no
config add ~/.zshrc && config commit -m "Shell-Konfiguration"

Nichts wird automatisch eingesammelt; jede Datei kommt einzeln per add dazu — dasselbe Allowlist-Prinzip, nur prozedural. Werkzeuge wie GNU Stow oder chezmoi bauen das Muster aus; für den Einstieg braucht es sie nicht (und sie wurden für diese Notizen nicht selbst durchlaufen — deshalb hier nur als Zeiger).

Geheimnisse: drei Verteidigungslinien

  • Allowlist — strukturell verhindern, dass Unbedachtes hineinrutscht.
  • Lesen, was committet wird. Konkret: settings.json kann einen env-Block mit API-Keys enthalten — vorher prüfen, git diff --cached vor jedem Commit.
  • Privates Repo — aber „privat" ist keine Sicherheitsmaßnahme. Es schützt vor fremden Blicken, nicht vor dem eigenen versehentlichen Öffentlichmachen später. Geheimnisse gehören gar nicht hinein.

Wenn doch eines gepusht wurde: rotieren, nicht nur löschen. Ein aus der Historie entferntes Secret kann längst kopiert sein. History-Umschreiben ist Kosmetik — der einzige sichere Zustand ist, dass das alte Token ungültig ist.

Der Restore-Test

Ein Backup, das nie zurückgespielt wurde, ist eine Vermutung. Bei Dotfiles ist der Test so billig, dass es keine Ausrede gibt: in ein Testverzeichnis klonen, prüfen, ob die Umgebung funktioniert. Für ~/.claude/: Sieht eine frische Claude-Code-Session die Skills und die CLAUDE.md? Erst danach ist es ein Backup.

Die Kernidee. Konfiguration ist Wissen, und Wissen braucht einen Ort, der einen Plattentod überlebt. Versioniert wird per Allowlist — weil die Verbotsliste immer eine Datei zu spät kommt.
