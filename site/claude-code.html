Claude Code

Kontext ist endlich

Fast jede Design-Entscheidung in Claude Code lässt sich auf einen einzigen Satz zurückführen. Wer ihn verstanden hat, trifft die Konfigurationsentscheidungen von selbst richtig.

Das Grundproblem

Das Kontextfenster ist der Arbeitsspeicher: alles, was Claude gerade sieht. Es ist begrenzt, und alles darin kostet — bei jeder einzelnen Anfrage neu.

Daraus folgt die Frage, die jeden Baustein unten erklärt: Muss das immer geladen sein, oder nur, wenn es gebraucht wird?

CLAUDE.md — die Projekt-Fakten

Wird bei jeder Anfrage mitgeladen. Mehrere Ebenen greifen zusammen:

DateiGilt fürCommittet?
~/.claude/CLAUDE.mddich, überallnein (liegt außerhalb)
<projekt>/CLAUDE.mdalle im Repoja
<projekt>/CLAUDE.local.mdnur dich, dies Reponein (auto-ignoriert)

Hinein gehört: Build-Befehle, Konventionen, Fallstricke. Nicht hinein: mehrstufige Prozeduren (→ Skills), und nichts, was aus dem Code selbst ablesbar ist.

Faustregel: unter ~200 Zeilen. Eine aufgeblähte CLAUDE.md ist eine Dauersteuer auf jede Anfrage, die du je stellst. Das ist auch der Grund, Regeln nicht vorsorglich zu sammeln: Zu früh gesetzte Regeln werden später ignoriert — und ignorierte Regeln untergraben das Vertrauen in alle anderen.

Permissions — was ohne Rückfrage laufen darf

DateiGilt fürCommitten?
~/.claude/settings.jsondich, alle Projekte–
.claude/settings.jsonalle im Repoja
.claude/settings.local.jsonnur dich, dies Reponein

Vorrang: local → project → user. Drei Listen, und die Aufteilung ist die eigentliche Lehre:

"allow": ["Bash(git status:*)"]        // Lesen: keine Rückfrage
"ask":   ["Bash(git push:*)"]         // Wirkung nach außen: fragen
"deny":  ["Bash(git push --force:*)"] // Unumkehrbar: nie

allow fürs Lesen · ask für Wirkung nach außen · deny für Unumkehrbares

Der Sinn ist nicht Bequemlichkeit. Wer alles durchwinkt, liest die Rückfragen irgendwann nicht mehr — und übersieht die eine, die zählte. Rückfragen sparsam einsetzen heißt, dass sie wieder Signal tragen.

Permission-Modi (Shift+Tab wechselt)

  • default — fragt bei Edits und Shell-Befehlen
  • plan — nur lesen, keine Änderungen
  • acceptEdits — Datei-Edits ohne Rückfrage
  • bypassPermissions — keine Rückfragen. Nur in Container/Sandbox vertretbar

plan ist der unterschätzteste: erst verstehen lassen, dann bauen. Fehler im Plan sind billig, Fehler im Code teuer.

Skills — Prozeduren, die nur bei Bedarf laden

.claude/skills/<name>/SKILL.md (Projekt) oder ~/.claude/skills/ (überall).

Fakten → CLAUDE.md. Prozeduren → Skill.

Das ist der ganze Unterschied: CLAUDE.md ist immer geladen. Ein Skill nur, wenn er relevant ist — er darf deshalb ausführlich sein, ohne zu kosten. Das description-Feld ist kein Kommentar, sondern der Auslöser: Es entscheidet, ob Claude den Skill überhaupt findet.

Vertiefung: Skills haben inzwischen eine eigene Seite — Aufbau, Frontmatter, Argumente, Fallstricke. Dort steht auch, was sich seit diesen Notizen geändert hat: Slash-Commands und Skills sind mittlerweile ein System.

Subagents — eigener Kontext für Teilaufgaben

.claude/agents/<name>.md — eine flache Datei, nicht <name>/AGENT.md. 

---
name: quick-reviewer          # PFLICHT: kleingeschrieben, Bindestriche
description: Scans code and suggests readability improvements   # PFLICHT: der Auslöser
tools: Read, Grep             # optional — hier: lesen, nicht schreiben
model: haiku                  # optional
---

You are a code improvement specialist. For each issue, explain the
problem and show an improved version. Keep feedback concise.

Der Markdown-Body ist der System-Prompt. Aufruf: automatisch (Claude entscheidet per description), per Nennung im Prompt, oder garantiert per @agent-<name>.

Warum das nützt: Ein Subagent hat ein eigenes Kontextfenster. Eine breite Suche liest fünfzig Dateien — im Hauptkontext wäre der danach voll. Als Subagent kommt nur das Ergebnis zurück.

Die Recherche für diese Notizen lief genau so: mehrere Subagents parallel für Claude-Code-Doku, GitHub-Pages-Doku und Nutzungsbedingungen. Zurück kamen Zusammenfassungen statt dutzender Doku-Seiten. Ein Subagent fand dabei auch den Fehler eines anderen: Das Agent-Format war nicht <name>/AGENT.md, wie zunächst vermutet.

Hooks — Automatik, die nicht vergessen wird

In settings.json unter "hooks". Der entscheidende Punkt: Die Umgebung führt sie aus, nicht das Modell.

Events u. a.: SessionStart, UserPromptSubmit, PreToolUse (kann blockieren), PostToolUse, Stop, PreCompact.

Eine Bitte kann übersehen werden. Ein Hook nicht.

Was wirklich immer gelten muss, gehört deshalb in einen Hook — nicht in eine Anweisung. Typisch: Formatter nach jedem Edit, Kontext beim Sitzungsstart laden, geschützte Dateien blocken.

Kontext-Management

  • /context — Auslastung ansehen. Bei Trägheit zuerst hier schauen
  • /compact [fokus] — Verlauf zusammenfassen, Platz schaffen
  • /clear — frische Session. CLAUDE.md bleibt, Verlauf ist weg
  • Auto-Compact ab ca. 85 % Auslastung

Faustregel: Neue Aufgabe → /clear. Kontext aus einer erledigten Aufgabe hilft der nächsten nicht, kostet aber weiter.

Was die Sitzung überlebt — und was nicht

/clear, Compaction und ein Rechnerausfall haben dieselbe Wirkung, nur in unterschiedlichem Tempo:

Die Konversation ist kein Asset. Sie ist Arbeitsspeicher.

Wissen, das nur im Chat existiert, ist bereits verloren — nicht erst beim Festplattentod, sondern beim nächsten /clear. Der Crash macht nur schlagartig sichtbar, was ohnehin ständig passiert.

Deshalb CLAUDE.md. Deshalb docs/. Deshalb ausführliche Commit-Messages. Nicht aus Ordnungsliebe — sondern weil das die einzigen Orte sind, an denen Wissen die Sitzung überlebt.

Den Arbeitsstand zwischen Sitzungen trägt Git selbst — sofern in sauberen Einheiten gearbeitet wird: Branch → PR → Merge, ausführliche Commit-Messages mit dem Warum (was gemacht wurde, zeigt der Diff; warum, steht sonst nirgends), offene Punkte in Issues statt im Gedächtnis. Die Probe: „Wo sind wir stehen geblieben?" muss sich aus git log und den offenen PRs beantworten lassen.

Hier stand früher eine .claude/progress.md als Zwischenspeicher für den Arbeitsstand, per SessionStart-Hook eingespeist. Gestrichen, nachdem die Probe aufs Exempel gelang: Eine neue Session konnte den Stand vollständig aus Historie und PRs rekonstruieren — weil nichts halbfertig liegen geblieben war. Ein solcher Puffer lohnt nur, wenn oft mitten in der Arbeit abgebrochen wird.

!befehl im Terminal führt ihn aus und legt die Ausgabe in den Chat — ideal für Interaktives wie gh auth login, das Claude nicht selbst bedienen kann.

Wo Claude arbeitet

Projektordner\        ← hier wird gearbeitet. Hier ist die Versionierung.
    .git\             ← komplette Historie, lokal
        │
        │  push  ↓↑  pull
        ▼
   github.com/...     ← eine Kopie. Backup, Sync, CI, Hosting.

Claude Code arbeitet an den Dateien auf der Platte, nicht „auf GitHub“. Änderungen im Browser oder von einem anderen Rechner sind unsichtbar, bis jemand git pull macht — die klassische Quelle für „aber ich hab das doch geändert?!“. (Claude Code in der Cloud arbeitet anders: dort auf einer Kopie des Repos.)

Die Kernidee. Kontext ist endlich. Lade Fakten immer (CLAUDE.md), Prozeduren bei Bedarf (Skills), lagere Breitensuche aus (Subagents), erzwinge Unverzichtbares deterministisch (Hooks).

Das gilt unabhängig von Version und Projekt.
