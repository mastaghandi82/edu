Claude Code · Vertiefung

Aufgeschriebene Erfahrung
mit einem Auslöser dran

Ein Skill ist eine Prozedur als Datei: Er lädt nur, wenn er gebraucht wird — und genau das macht ihn zum Gegenstück der immer teuren CLAUDE.md.

Wofür Skills gut sind

Wer Claude dreimal erklärt hat, wie in diesem Projekt deployt wird, schreibt es beim vierten Mal als Skill auf. Ab dann kennt jede Session den Ablauf — auch die von Kolleg:innen, denn Projekt-Skills werden committet. Das löst zwei Probleme auf einmal: Wiederholung (die Erklärung existiert genau einmal) und Kontextkosten (sie kostet nur, wenn sie gebraucht wird).

Fakten → CLAUDE.md. Prozeduren → Skill.

Konventionen, Build-Befehle, Eigenheiten: CLAUDE.md. Mehrstufige Abläufe, Checklisten, Referenzmaterial, Workflows mit Nebenwirkungen: Skill.

Stand der Dinge: Eigene Slash-Commands und Skills waren früher zwei Mechanismen — inzwischen sind sie ein System. Beide erzeugen /<name>; alte .claude/commands/-Dateien laufen weiter, aber Skills sind der empfohlene Weg: ein ganzes Verzeichnis statt einer Datei, feinere Steuerung, automatisches Laden. (Die Hauptseite zu Claude Code beschreibt noch den älteren Stand mit zwei Bausteinen.)

Die Kernmechanik: gestaffeltes Laden

Der Grund, warum Skills fast nichts kosten, heißt Progressive Disclosure — Wissen wird in Stufen sichtbar:

StufeWas ist im KontextKosten
immernur name + description~100 Tokens pro Skill
beim Auslösender komplette Skill-Texteinmalig, bleibt für die Session
bei BedarfZusatzdateien, Skriptenull, bis Claude sie liest

Das erklärt zwei Dinge sofort:

  • Warum die description alles entscheidet. Sie ist das Einzige, was Claude sieht, bevor der Skill geladen ist. Vage Beschreibung = ein Skill, der nie feuert.
  • Warum ein Skill ausführlich sein darf. Der Inhalt kostet erst beim Auslösen — anders als jede Zeile CLAUDE.md, die eine Dauersteuer ist.

Änderungen an einer SKILL.md wirken sofort, ohne Neustart. In Monorepos werden auch .claude/skills/ in Unterverzeichnissen gefunden.

Aufbau: ein Verzeichnis, eine Pflichtdatei

deploy/
├── SKILL.md           ← Pflicht: Anleitung + Frontmatter
├── reference.md       ← Detailwissen, erst bei Bedarf gelesen
└── scripts/
    └── check.sh       ← ausführbare Helfer

Faustregel: SKILL.md unter ~500 Zeilen, Detailtiefe in Zusatzdateien. Ein vollständiges Minimalbeispiel: 

---
description: Deployt site/ nach GitHub Pages und verifiziert die
  Live-URL. Nutzen bei "deploy", "veröffentlichen", "live stellen".
disable-model-invocation: true
---

1. Sicherstellen, dass auf einem Feature-Branch gearbeitet wird.
2. PR erstellen, mergen — der Workflow deployt bei Push auf main.
3. Lauf beobachten: gh run list → ID → gh run watch <id> --exit-status
4. Erst fertig melden, wenn die Live-URL die Änderung zeigt —
   ein grüner Actions-Lauf allein reicht nicht.

Wer darf auslösen?

Zwei Wege führen zum selben Ergebnis: automatisch (Claude erkennt an der description, dass der Skill passt) oder manuell per /skill-name argumente. Zwei Frontmatter-Felder steuern, welcher Weg offen ist — und das ist die eigentliche Design-Entscheidung pro Skill:

  • disable-model-invocation: true — nur manuell. Für alles mit Nebenwirkungen (deployen, pushen, löschen): der Mensch bleibt der Auslöser, Claude startet so etwas nicht „bei Gelegenheit".
  • user-invocable: false — umgekehrt: nicht im /-Menü, nur Claude nutzt ihn. Für reines Wissen, das wirken soll, ohne dass man dran denkt.

Im Skill-Text stehen Platzhalter zur Verfügung: $ARGUMENTS (alles), $0, $1, … (einzeln), ${CLAUDE_SKILL_DIR} (Pfad zum Skill-Verzeichnis — für Hilfsskripte). Und mit !`befehl` läuft ein Shell-Befehl beim Laden, dessen Ausgabe in den Text eingesetzt wird, bevor Claude ihn sieht: Die Prozedur startet mit frischen Fakten statt mit Annahmen.

Gute Skills schreiben

  • Die description ist das Produkt. Sie muss sagen, was der Skill tut und wann er dran ist — mit den Wörtern, die in echten Aufgaben vorkommen, nicht mit Binnenjargon.
  • Eine Prozedur pro Skill. „Alles rund um Git" triggert nie richtig. Lieber drei kleine als einen breiten.
  • Schreiben wie für neue Kolleg:innen: nummerierte Schritte, konkrete Befehle, erwartete Ergebnisse, bekannte Fallen.
  • Testen wie ein Feature: frische Session; eine Aufgabe, die den Skill auslösen sollte — feuert er? Eine, die ihn nicht auslösen sollte — bleibt er still? Anthropics Plugin skill-creator automatisiert das bis hin zum Messen der Trigger-Rate.

Der klassische Denkfehler: alles in die CLAUDE.md kippen. Wenn Claude eine Regel ignoriert, ist die CLAUDE.md meist zu lang — die Antwort ist nicht mehr Text dort, sondern auslagern in Skills.

Über Claude Code hinaus

Skills gibt es auch im Agent SDK (liest dieselben Dateien aus dem Dateisystem), auf claude.ai (ZIP-Upload) und in der API (Upload per /v1/skills). Aber: sie synchronisieren nicht. Ein Skill ist portables Format, kein geteilter Speicher — wer ihn auf zwei Flächen will, pflegt ihn zweimal.

Die Kernidee. Ein Skill ist aufgeschriebene Erfahrung mit einem Auslöser dran. Die description entscheidet, ob er gefunden wird; gestaffeltes Laden sorgt dafür, dass Wissen erst kostet, wenn es gebraucht wird.

Wer eine Erklärung zum dritten Mal tippt, hat einen Skill identifiziert.
